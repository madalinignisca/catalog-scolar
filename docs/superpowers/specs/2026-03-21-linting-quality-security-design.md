# Linting, Code Quality & Security Checks — Design Spec

**Date:** 2026-03-21
**Status:** Draft
**Scope:** Pre-commit hooks + local tooling for all checks. CI integration deferred.

## Overview

Set up comprehensive linting, code quality, and security scanning for the CatalogRO monorepo, enforced via pre-commit hooks. Covers Go API, Nuxt frontend, SQL migrations, Dockerfiles, Helm charts, dependency auditing, and secret detection.

## Design

### 1. Go Linting — `.golangci.yml`

Create `api/.golangci.yml` with the following linters enabled:

**Bug catchers:**
- `govet` — compiler-level correctness (printf formats, struct tags)
- `staticcheck` — the gold standard Go analyzer (unreachable code, deprecated APIs, incorrect sync usage)
- `errcheck` — unchecked error returns
- `nilerr` — returning nil error when err is non-nil
- `bodyclose` — unclosed HTTP response bodies
- `sqlclosecheck` — unclosed `sql.Rows` / `sql.Stmt`
- `exhaustive` — non-exhaustive switch on enum types (qualifier, role, semester enums)

**Security:**
- `gosec` — OWASP-style checks (hardcoded creds, weak crypto, SQL injection)

**Code quality:**
- `revive` — configurable superset of golint (naming, unexported returns, blank imports)
- `gocritic` — opinionated style + performance (unnecessary type conversions, append patterns)
- `gosimple` — simplification suggestions
- `ineffassign` — assignments to variables never read
- `unused` — unused code detection

**Excluded:**
- `gocognit`/`cyclop` — complexity limits too noisy for a project at 0% implementation
- `wrapcheck` — error wrapping enforcement creates friction during rapid development
- `varnamelen` — too opinionated on variable naming

**Configuration notes:**
- Timeout: 5 minutes (generous for CI, irrelevant for hooks which scope to changed files)
- Exclude test files from gosec rule G104 (unhandled errors in tests are acceptable)
- Exclude generated code: path `db/generated/` (relative to `api/`, since `.golangci.yml` lives inside `api/` and the Makefile runs `cd api && golangci-lint run ./...`)

### 2. Web Linting — ESLint Additions

Add to existing ESLint flat config:

**New plugins:**
- `eslint-plugin-security` — dangerous function usage, RegExp DoS, prototype pollution, unsafe innerHTML, object injection
- `eslint-plugin-vuejs-accessibility` (>=2.5.0 required for flat config support) — missing alt, non-interactive click handlers, missing form labels. Import as `import vuejsAccessibility from 'eslint-plugin-vuejs-accessibility'` and spread `...vuejsAccessibility.configs['flat/recommended']`.
- `eslint-plugin-import-x` — broken imports, import order, circular dependency detection

**New strict rules:**
- `@typescript-eslint/no-floating-promises: 'error'` — uncaught async errors (critical for offline sync code)
- `@typescript-eslint/strict-boolean-expressions` with options `{ allowNullableObject: true, allowNullableBoolean: true, allowNumber: true, allowNullableString: false }` — prevents truthy/falsy bugs while allowing idiomatic Vue patterns like `v-if="someObj"` and `if (array.length)`
- `vue/no-v-html: 'error'` — prevents XSS via v-html

**Not adding:**
- `eslint-plugin-prettier` — Prettier already runs separately
- `eslint-plugin-tailwindcss` — Prettier plugin handles class sorting
- `eslint-plugin-unicorn` — too opinionated

**Typed linting setup:**
Requires `typescript-eslint` typed config for `no-floating-promises` and `strict-boolean-expressions`. Use `parserOptions.projectService: true` (typescript-eslint v8 API). This depends on `.nuxt/tsconfig.json` existing, which is generated by `nuxt prepare`. Since `package.json` has `"postinstall": "nuxt prepare"`, running `npm install` creates it. Fresh clones that skip `npm install` will see ESLint errors — this is expected and documented in the Makefile `hooks-install` target.

### 3. Pre-commit Hooks

Using the `pre-commit` framework (Python-based, manages tool installation and caching).

**Hook execution order** (fast to slow, fail-fast):

| # | Hook | Type | Scope | ~Time | Purpose |
|---|------|------|-------|-------|---------|
| 1 | gitleaks | repo: `gitleaks/gitleaks` | all staged files | <1s | Secrets in committed code |
| 2 | editorconfig-checker | repo: `editorconfig-checker/editorconfig-checker` | all staged files | <1s | Encoding, whitespace, line endings |
| 3 | hadolint | repo: `hadolint/hadolint` (via Docker or binary) | `Dockerfile` files | <1s | Dockerfile anti-patterns |
| 4 | prettier | local | `web/**/*.{ts,vue,json,css}` | ~1s | Formatting |
| 5 | eslint | local | `web/**/*.{ts,vue}` | ~2s | TS/Vue bugs, security, a11y |
| 6 | golangci-lint | local | triggers on `api/**/*.go` changes | ~3s | Go bugs, security, style |
| 7 | govulncheck | local | triggers on `api/go.sum` change | ~2s | Known CVEs in Go deps |
| 8 | npm audit | local | triggers on `web/package-lock.json` change | ~1s | Known CVEs in npm deps |
| 9 | helm lint | local | triggers on `helm/**` changes | ~1s | Chart template errors |
| 10 | semgrep | local | `api/**/*.go` | ~3s | SQL injection, RLS bypass, crypto |

**Implementation details for non-trivial hooks:**
- **golangci-lint (hook #6):** Must use `pass_filenames: false` with `entry: bash -c 'cd api && golangci-lint run ./...'` and `files: '^api/.*\.go$'` to trigger only on Go file changes. golangci-lint requires full module context and cannot lint individual files.
- **helm lint (hook #9):** Local hook type with `entry: helm lint helm/catalogro` and `files: '^helm/'`. No official pre-commit repo exists for helm.
- **govulncheck (hook #7):** Local hook with `entry: bash -c 'cd api && govulncheck ./...'` and `files: '^api/go\.sum$'`.
- **npm audit (hook #8):** Local hook with `entry: bash -c 'cd web && npm audit --audit-level=high'` and `files: '^web/package-lock\.json$'`.

**Design decisions:**
- Gitleaks first — if secrets are staged, stop immediately
- Formatting (prettier) before logic (eslint) — auto-fixable first
- `govulncheck` and `npm audit` trigger only on dependency file changes
- Each hook scoped to its directory — Go changes don't trigger web hooks

**Estimated total time:** ~10-14s for commits touching both stacks, ~5-7s for single-stack.

**Dropped:** sqlfluff. Migrations are goose-managed and sqlc-validated. sqlfluff adds ~4s and conflicts with goose markers. SQL correctness is better validated by `make sqlc` + `make migrate`.

### 4. Semgrep — Focused Go Security Rules

Custom `.semgrep.yml` at repo root. Not using semgrep's full registry.

**Rules (7):**

| Rule ID | Severity | Pattern | Feasibility |
|---------|----------|---------|-------------|
| `sql-string-concat` | ERROR | `fmt.Sprintf` or `+` building SQL queries | Reliable — single-expression AST match |
| `rls-missing-tenant-context` | WARNING | Direct `pool.Query`/`pool.Exec` call in HTTP handler functions without `SetTenant` in the same function body | Best-effort — Semgrep does intra-function analysis only. Cannot trace middleware calls. Will need `// nosemgrep` on legitimate cases where tenant context is set by middleware. |
| `hardcoded-school-id` | WARNING | UUID literals resembling school IDs in non-test code | Reliable — pattern match on UUID format in assignments |
| `jwt-secret-in-code` | ERROR | String literals assigned to JWT/TOTP secret variables | Reliable — variable name + string literal pattern |
| `unvalidated-user-input-to-db` | WARNING | HTTP request body/params flowing to DB query params within the same function | Best-effort — limited to intra-function taint tracking (Semgrep OSS limitation). Will miss cases where handler and DB call are in separate functions. Inter-file taint requires Semgrep Pro. |
| `insecure-crypto` | ERROR | Import of `crypto/md5`, `crypto/sha1`, `crypto/des` packages | Reliable — import statement match |
| `missing-error-check-after-rows` | WARNING | `db.Query()` without checking `rows.Err()` after iteration | Reliable — intra-function pattern |

**Scoping:**
- Only scans `api/**/*.go`
- Test files (`*_test.go`) excluded from `rls-missing-tenant-context` and `hardcoded-school-id`

### 5. Supporting Config Files

**`.editorconfig`** (repo root):

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true
indent_style = space
indent_size = 2

[*.go]
indent_style = tab

[Makefile]
indent_style = tab

[*.sql]
indent_size = 4

[*.md]
trim_trailing_whitespace = false
```

- Go uses tabs (gofmt convention)
- SQL at 4-space indent (matches existing migration style)
- LF line endings enforced (prevents CRLF issues in SIIIR parser)

**`.gitleaks.toml`** (repo root) — allowlist for known false positives:

Allowlist the localhost database credentials in `Makefile` and `CLAUDE.md` (e.g., `postgres://catalogro:catalogro@localhost:5432/...`). These are development-only values, not secrets. Also allowlist `api/db/seed.sql` which contains stable test UUIDs that may trigger generic secret patterns.

**`web/.prettierignore`:**
```
.nuxt/
.output/
node_modules/
dist/
*.min.js
```

### 6. Makefile Additions

| Target | Description |
|--------|-------------|
| `make check` | Run all checks in hook order (manual, without committing) |
| `make security` | gosec + govulncheck + npm audit + gitleaks + semgrep |
| `make fix` | Auto-fix: prettier --write, eslint --fix, goimports -w (goimports handles both formatting and import grouping) |
| `make hooks-install` | Install pre-commit framework + run `pre-commit install` |

- `npm install` (triggers `nuxt prepare` via postinstall) must run before ESLint typed linting works, since `parserOptions.projectService` depends on `.nuxt/tsconfig.json`

## Not in Scope

- CI/CD workflow updates (deferred per user decision)
- sqlfluff (conflicts with goose markers, low ROI)
- License compliance checking (better suited to CI)
- OpenAPI/Spectral linting (no spec file exists yet)
- gocognit/cyclop complexity limits (premature for 0% implementation)
- eslint-plugin-unicorn (too opinionated)
- Semgrep Pro inter-file taint analysis (two rules noted as best-effort with OSS engine)
